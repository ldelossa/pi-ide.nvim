INTRODUCTION                                                     *pi-ide-intro*

`pi-ide.nvim` is a Neovim plugin that serves the `pi-ide` protocol over a
local WebSocket. External AI agents like `pi` and `claude-code` connect to
the plugin to open diffs in Neovim, read LSP diagnostics, list open buffers,
receive cursor and selection notifications, and request inline code
completions (often called "suggestions") that render as ghost text.

This is the reference editor implementation for the `pi-ide` Pi extension at
<https://github.com/ldelossa/pi-ide>. The `pi-ide` extension expects an
editor that speaks this protocol, and `pi-ide.nvim` implements this in
Neovim.

The plugin is opinionated about where the agent runs. It assumes `pi` or
`claude-code` are running as external processes, outside of Neovim. The
plugin does not embed an agent, does not call out to an LLM directly, and
does not hold any prompt state. The inline suggestion feature is the one
place where an LLM is involved, but the model call is made by the connected
`pi-ide` extension running inside `pi`; Neovim only gathers context and
renders ghost text.

SETUP                                                            *pi-ide-setup*

Install via your favorite plugin manager.

Using lazy.nvim:
>
    {
        "ldelossa/pi-ide.nvim",
        config = function()
            require("pi-ide").setup()
        end,
    }
<

The default configuration is:
>
    require("pi-ide").setup({
        auto_start = true,                 -- start the server on plugin load
        claude_code_compatibility = false, -- write claude-code lockfile too
        log_level = "warn",                -- trace, debug, info, warn, error
        suggestion = {
            auto_trigger = true,           -- debounced fire on TextChangedI
            default_keys = true,           -- install <M-\>, <M-]>/<M-[>, <Tab>, <C-]>
            model = nil,                   -- optional. preferred model "provider/id"
        },
    })
<

COMMANDS                                                      *pi-ide-commands*

Six user commands are installed.

                                                                     *:PiStart*
`:PiStart`              Start the MCP server on a random free port and write
                        the lockfile.

                                                                      *:PiStop*
`:PiStop`               Stop the server, close all connected clients, and
                        reject any open diffs.

                                                                    *:PiStatus*
`:PiStatus`             Open a floating window showing the server port,
                        connected client count, and lockfile path.

                                                                   *:PiSuggest*
`:PiSuggest`            Manually trigger an inline suggestion at the cursor.
                        Requires a connected `pi-ide` extension client and an
                        active LSP plus treesitter parser for the current
                        buffer.

                                                             *:PiSuggestToggle*
`:PiSuggestToggle`      Toggle automatic (debounced) suggestion triggering on
                        or off for the current session. The manual command
                        keeps working when auto is off.

                                                                *:PiSuggestModel*
`:PiSuggestModel`       Ask the connected `pi-ide` extension for available
                        models and select the runtime suggestion model for
                        this Neovim session.

CLAUDE CODE COMPATIBILITY                                  *pi-ide-claude-code*

`pi-ide.nvim` is wire compatible with the `claude-code` IDE integration
protocol. To enable it, set `claude_code_compatibility = true` in your setup
call:
>
    require("pi-ide").setup({
        claude_code_compatibility = true,
    })
<

When enabled, the plugin writes a second lockfile to
`~/.claude/ide/<port>.lock` alongside the usual `~/.pi/ide/<port>.lock`, and
the handshake accepts either the `x-pi-ide-authorization` or
`x-claude-code-ide-authorization` header. From `claude-code`'s perspective
the plugin appears as a standard IDE integration and discovery just works.

==============================================================================
TOOLS                                                            *pi-ide-tools*

Four tools are exposed over the MCP `tools/call` method.

                                                            *pi-ide-openDiff*
`openDiff`              Open a diff tab between an existing file and proposed
                        contents. A new tabpage opens with the existing file
                        on the left and the proposed contents on the right.
                        Edit the right buffer freely, then save with `:w` to
                        accept (optionally with your own edits applied), or
                        close either window to reject. The JSON-RPC response
                        is held open until the diff resolves.

                                                           *pi-ide-close_tab*
`close_tab`             Close a previously opened diff tab by name.

                                                      *pi-ide-getDiagnostics*
`getDiagnostics`        Return LSP diagnostics for a single file (by URI) or
                        for all loaded buffers if no URI is given.

                                                   *pi-ide-getOpenEditorTabs*
`getOpenEditorTabs`     List the buffers visible in the current tabpage.

==============================================================================
NOTIFICATIONS                                            *pi-ide-notifications*

The server pushes a single notification, `selection_changed`, on cursor
movement, mode change, buffer entry, and text change. The notification is
debounced at 100ms to avoid flooding the client during rapid editing. The
payload includes the current file, cursor position, and selected range and
text if any.

==============================================================================
SUGGESTIONS                                                *pi-ide-suggestions*

Inline ghost-text suggestions, similar in style to GitHub Copilot but routed
through the `pi-ide` Pi extension. The editor initiates the request; the
extension calls an LLM and returns up to three alternatives; Neovim renders
them as virtual text and handles cycling and partial acceptance.

Suggestions always work regardless of editor setup. For the best
results, make sure both an LSP client and a treesitter parser are active
for the buffer — they provide structural and diagnostic context that
improves completion quality. When either is missing, the feature still
works but with reduced context.

Setup options under `suggestion` are passed through to the suggestion
module:
>
    require("pi-ide").setup({
        suggestion = {
            auto_trigger = true,    -- debounced on TextChangedI
            default_keys = true,    -- install <M-\>, <M-]>/<M-[>, <Tab>, <C-]>
            model = nil,            -- optional. preferred model "provider/id"
        },
    })
<
Model precedence: the `pi` CLI flag `--pi-ide-suggestion-model
<provider>/<id>` wins. Otherwise the editor-provided `model` setup option
is sent in the request and used. If neither is set, the current session's
model is used.

To find valid `provider/id` strings, run `pi --list-models` and join the
`provider` and `model` columns with a `/` (e.g., `openai/gpt-4o`). You can
also run |:PiSuggestModel| while connected to pi to pick an available model
for the current Neovim session. If pi was started with
`--pi-ide-suggestion-model`, that CLI override still wins.

For suggestions, prefer low-latency models. Large reasoning models often
feel too slow for inline completion; smaller coding-capable models such as
`openai-codex/gpt-5.4-mini` or `deepseek/deepseek-v4-flash` usually provide
a better interactive experience.

Default insert-mode keys (set `default_keys = false` to skip):

    <M-\>         manually trigger a suggestion (|<Plug>(PiSuggest)|)
    <M-]>         cycle to next alternative (|<Plug>(PiSuggestNext)|)
    <M-[>         cycle to previous alternative (|<Plug>(PiSuggestPrev)|)
    <Tab>         accept the full suggestion (|<Plug>(PiSuggestAccept)|)
    <C-]>         dismiss the active suggestion (|<Plug>(PiSuggestDismiss)|)

These mirror copilot.vim's canonical defaults so users migrating from
copilot.vim can drop in pi-ide.nvim without remapping. Accept-line and
accept-word are exposed as |<Plug>(PiSuggestAcceptLine)| and
|<Plug>(PiSuggestAcceptWord)| but unbound by default; bind them yourself
to keys that fit your terminal and existing setup.

Terminal caveats:

  - `<Tab>` is the default for accept and often conflicts with snippet
    plugins, completion menus, or any super-tab logic the user already
    has. If you use snippets or a tab-driven completion popup, set
    `default_keys = false` and bind |<Plug>(PiSuggestAccept)| to a
    different key.

Display gating: suggestions are suppressed on lines where the inline
virt_text would render in the wrong screen position — specifically when
`conceallevel > 0` on the current window, or when the current line is
wide enough to wrap inside the window. Auto-triggers skip silently;
manual `:PiSuggest` emits a warning so you know why nothing appeared.

Session lifecycle: once a suggestion arrives, typing characters that match
the suggestion advances the ghost text without firing a new LLM call.
Typing a character that diverges first tries to cycle to an alternative
whose prefix matches; if none match, the session is dismissed and the next
debounce will fire a fresh request. Partial acceptance preserves the
remaining tail of the suggestion as a new ghost-text session, so a single
LLM call can drive multiple word- or line-sized accepts.

ARCHITECTURE                                              *pi-ide-architecture*

`pi-ide.nvim` runs a single MCP server per Neovim instance, listening on a
random free port on `127.0.0.1`. The server speaks JSON-RPC 2.0 in WebSocket
text frames using MCP protocol version `2024-11-05`. Tool calls flow from
the connected agent into Neovim; the plugin also initiates its own requests
(`getSuggestions` and `listSuggestionModels`) back to the agent for the
inline suggestion feature.

On startup the plugin writes a lockfile to `~/.pi/ide/<port>.lock`. Clients
discover the server by reading lockfiles in this directory and matching the
`workspaceFolders` field against their `cwd`. A random per-session auth
token gates access. The token is included in the lockfile and validated on
every connection.

FILE LAYOUT                                                      *pi-ide-files*

>
    lua/pi-ide/
        init.lua          -- public API and user commands
        lockfile.lua      -- auth token generation, lockfile create/remove
        logger.lua        -- vim.notify wrapper with log levels
        selection.lua     -- cursor and selection tracking
        diff.lua          -- diff tab lifecycle and coroutine accept/reject
        tools.lua         -- MCP tool definitions and dispatch
        suggestion/
            init.lua      -- suggestion session state machine and autocmds
            context.lua   -- treesitter outline, enclosing scope, cursor window
            render.lua    -- virtual text rendering via extmarks
            keys.lua      -- <Plug> mappings and default insert-mode keymaps
        server/
            init.lua      -- JSON-RPC dispatch and deferred response routing
            tcp.lua       -- libuv accept loop, client registry, ping timer
            client.lua    -- per-client state and WebSocket frame loop
            frame.lua     -- WebSocket frame parse and emit
            handshake.lua -- HTTP Upgrade handling and auth header check
            utils.lua     -- base64, SHA-1, UTF-8, byte packing, XOR mask
<

The WebSocket implementation under `server/` is roughly 1200 lines of pure
Lua. It exists because Neovim ships no WebSocket library and the Lua
ecosystem has no de facto standard. If you ever swap the transport,
`frame.lua`, `handshake.lua`, and most of `utils.lua` go away.
